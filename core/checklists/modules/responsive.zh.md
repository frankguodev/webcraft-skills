# Responsive High-Risk Module

用于补强 `ui-audit.zh.md` 中响应式稳定性的检查，重点发现“桌面看着正常，但关键视口一打开就崩”的问题。

## 使用时机

- 审查页面、整站、后台模块、表格/卡片列表、富文本、搜索筛选、表单、弹窗或首屏 hero。
- 页面有固定宽度、横向排列、复杂 grid、长标题、图片/mockup、表格、代码块、图表或 sticky 区域。
- 用户反馈移动端、平板或某个窗口宽度下出现压缩、空洞、错位、横向滚动。

## 必查点

- 关键视口：至少检查或推断 `375px`、`768px`、`1280px`；Deep Audit 再扩大到小手机、大手机、宽屏。
- 横向滚动：body、section、表格、代码块、图片、mockup、按钮组、chip 组都不能造成非预期横向滚动。
- 降级策略：桌面多列、左右分栏、卡片网格、工具栏、筛选器、表格在小屏和平板要有明确折叠、换行、滚动或重排策略。
- 稳定尺寸：图片、视频、iframe、图表、截图、卡片、按钮、工具栏应使用稳定比例、min/max、容器约束或合理换行。
- 中间断点：平板、小桌面、浏览器半屏不能出现比手机或桌面更差的尴尬状态。
- sticky/fixed：顶部导航、底部操作条、侧栏、筛选栏不能遮挡内容、表单错误、弹窗或锚点目标。

## 浏览器验证

- 实测时不要只看最终截图；拖动窗口或切换几个宽度，观察布局是否突然跳变、空洞或挤压。
- 检查 body 是否出现横向滚动条，必要时定位具体溢出元素。
- 对长文案、多标签、多卡片、表格列数、不同图片比例做最小内容压力测试。

## 典型问题信号

- 移动端只是桌面缩小版，文字和按钮可见但难读难点。
- 768px 或 834px 比手机端更乱，说明断点只照顾了桌面和手机。
- 图片或 mockup 没有 aspect-ratio，加载前后高度跳动。
- 表格、代码块、长筛选条件没有横向滚动容器或折叠策略。
- sticky header 遮住锚点标题，或底部操作条遮住表单提交/错误信息。

## 修复边界

- 优先修不可用、横向滚动、遮挡和内容裁切，再修 spacing 和视觉细节。
- 响应式修复应保持原信息架构，不随意删除业务信息。
- 不要靠 `overflow: hidden` 掩盖溢出；必须确认被隐藏内容不是核心信息或操作。
