# Accessibility High-Risk Module

用于补强 `ui-audit.zh.md` 中可访问性基线检查，重点发现会影响键盘、读屏、可见焦点、可点击目标和状态理解的问题。

## 使用时机

- 页面包含表单、弹窗、菜单、下拉、多选、图标按钮、卡片操作、表格、上传或危险操作。
- 用户反馈“键盘不好用”“看不出当前焦点”“图标按钮不知道什么意思”。
- 审查范围涉及上线前、后台管理、表单流程、移动端操作或高对比度/无障碍要求。

## 必查点

- 键盘路径：核心任务必须能用键盘完成，Tab 顺序应符合视觉和任务顺序。
- focus-visible：焦点必须可见，不能被全局 outline reset、overflow、遮罩或颜色对比问题隐藏。
- 可访问名称：图标按钮、菜单按钮、关闭按钮、上传按钮、卡片操作必须有文本、aria-label 或等价说明。
- 语义结构：heading 顺序、button/link 语义、label/input 关联、错误提示关联和表格语义要支持理解。
- 状态表达：错误、成功、禁用、当前、选中、loading 不能只靠颜色表达。
- 目标尺寸：移动端和触控场景下，核心操作目标不能过小或过密。
- 高对比度：forced colors / high contrast 下，文本、边框、focus、图标和状态反馈仍应可辨认。

## 浏览器验证

- 用 `Tab` 走一遍核心路径，确认焦点顺序、焦点可见性、弹窗进入/退出和返回触发点。
- 对图标按钮和自定义控件检查可访问名称；能用浏览器 accessibility tree 时优先实测。
- 对弹窗、菜单、下拉、多选检查 Escape、关闭、焦点管理和键盘选择。

## 典型问题信号

- `outline: none` 后没有替代 focus-visible。
- 可点击卡片、图标按钮或关闭按钮没有可访问名称。
- 弹窗打开后焦点仍停留在背景，关闭后没有返回触发按钮。
- 错误只靠红色边框表达，读屏和视觉用户都不知道怎么修。
- 按钮或 chip 太小、太密，移动端误触风险高。

## 修复边界

- 可访问性问题不是视觉 polish；影响核心路径时应提升为 `Critical` 或 `Major`。
- 不要为了视觉干净隐藏 focus、label、帮助文本或错误信息。
- 优先复用项目已有组件库的可访问性能力，不要在单个页面手写零散补丁。
