# UI Fix 工作流

用于根据审查结果（review / audit ）直接修复 UI 代码。`fix-ui` 的目标是把已确认的问题修到可用、清晰、一致，而不是顺手重做整个页面。

默认优先修有证据、有影响、有明确修复方向的问题。允许同时修直接依赖、本次修改引入的新错误、以及不修就无法完成原问题的相邻问题；没有证据或需要产品决策的问题放入 `Open Questions`，不要假装已经确认。

## 语言规则

读取本 workflow 前，skill 的 Locale Contract 已经选择了一个运行语言。继续使用这个语言，不要再打开配对的英文或中文 workflow/checklist/module 文件，除非用户明确要求翻译、双语比较、本地化或语言一致性检查。

## 1. Baseline Gate / 修复前基线

开始改代码前，先确认修复边界和验证基线。如果无法完整确认，也要基于可获得信息建立最小基线；不要在完全不了解入口、验证方式和改动边界时直接大改：

- 修复来源和范围：audit、review、用户指定问题、截图或代码观察；单页、组件、整站、视口或状态。
- 风险类型：媒体、表格/富文本/代码块、滚动层、原生控件、动效、高对比度、页面入口、路由或数据渲染。
- 验证基线：项目已有 dev、lint、typecheck、test、build、preview 命令；受影响页面入口、localhost URL 或可复用运行页面。
- 相邻导航面：进入该页面的导航、菜单、列表、卡片、搜索结果或 CTA；同功能区列表/详情路由；离开该页面的返回、相关链接或核心操作。
- 保留边界：现有风格、主题方向、业务逻辑、真实数据和技术栈。

如果用户说“直接修”“帮我改好”，可以执行；如果范围可能很大，先给简短修复计划。

### Gate 深度按风险调整

`fix-ui` 的第一目标是修改问题的同时保证系统可用。

- 小范围样式修复：建立最小基线，运行可用验证命令，必要时做页面与路径验证。
- 影响页面入口、路由、组件逻辑、数据渲染、状态或交互的修复：必须做页面与路径验证。
- 涉及多文件、共享组件、布局断点、表单、弹窗、导航、原生控件替换的修复：必须做对应 UI 复检。
- 如果本次修改可能影响 runtime，即使 build 通过，也应打开受影响页面。
- 不要把所有 fix 都跑成完整路径矩阵；只有涉及页面入口、路由、导航、列表/详情、Next.js App Router、共享组件、数据渲染或客户端状态时，才必须做完整页面与路径验证。纯视觉小修通常只需要轻量工程验证和直接路由检查。

如果验证失败且无法在当前上下文修好，停止扩大改动，保留最小必要修改，并在输出中把失败命令、错误现象、已尝试修复和剩余阻塞写清楚。

## 2. 保留现有视觉体系

修复前应识别并尊重与本次改动相关的现有主题风格、框架、组件、样式体系和命名习惯，除非用户明确要求，不要为了修一个问题，把页面改成另一种风格。

保留主题不是冻结缺陷。只要某个视觉特征造成明确的可用性、可读性、响应式、状态、交互或可访问性问题，就必须修；但修复方式应在原主题内完成，例如校正、降噪、收敛、补状态、加约束、调整局部 token，而不是换成另一套视觉方案。

如果要从根本上改整体风格方向，必须先放入 `Open Questions` 让用户确认。

## 3. 根据修复来源选择策略

- `audit`：按 Critical / Major / Minor 和 Fix Order 修；如果来源是 `Focused Audit`，优先修 Top Findings、系统性 Major 和风险视口问题，不扩展成 Deep 级别重构。
- `review`：优先修 reviewer 明确指出的风险；方向性建议先转成可执行问题，不扩展成 redesign。
- `用户指定问题`：优先修点名问题和直接依赖；相邻问题只有在阻断原问题、导致页面打不开或由本次修改引入时才一并修，并说明原因。
- `截图`：只修可见布局、视觉、层级、文案；无法从截图确认的交互必须运行页面后再修。
- `代码观察`：标明推断来源；能运行页面时优先验证后修，不能运行时保守处理并说明未验证风险。

## 4. 需要用户决策的问题

以下问题不要擅自决定，放入 `Open Questions`：

- 产品定位、目标用户、核心卖点不清。
- 真实数据、案例、客户、价格、指标需要确认。
- 品牌色、字体、视觉方向需要取舍。
- 删除业务内容会影响信息完整性。
- 修复需要引入新依赖、换组件库或重构设计系统。
- 修复需要改变核心信息架构、数据模型、真实业务流程或删除重要内容。

## 5. Recheck Gate / 修复后复检

每次修复后，按被修问题复检对应维度，不需要展开无关矩阵：

- 如果修复来源包含 audit mode，复检优先覆盖该 audit 实际检查过的问题视口和风险视口；不要因为来源是 audit 就自动扩大到 Deep 级别矩阵。
- 布局/响应式：复检相关视口，尤其问题出现的宽度；首屏、搜索区、hero、mockup、section 间距、overflow、sticky/fixed、滚动和层级要确认没有新增遮挡、裁切或跳动。
- 媒体/富内容：复检图片、视频、iframe、图表、表格、代码块、富文本和长列表的比例、裁切、滚动、折叠和长内容。
- 组件/控件/交互：复检 hover、active、focus-visible、disabled、loading、error、点击热区，以及选择、下拉、多选、菜单、弹窗、导航的打开/关闭/键盘路径。
- 视觉/内容/可访问性：复检 token 一致性、主题方向、长文案和中英文混排、可访问名称、label/error 关联、forced colors / 高对比度和 reduced motion。
- 页面/路由：至少复检一个真实入口路径和目标路由；如果改到导航、菜单、列表项、卡片链接或共享组件，还要复检一个相邻出口、同级路由或核心操作。

如果无法运行页面，说明哪些复检只能基于代码推断。

## 6. Verification And Recovery Gate / 验证与失败回修

验证不是可选 polish。只要 fix 改了代码，就必须优先确认项目仍能运行；如果新错误来自本次修改，先修错误，再继续复检 UI。构建通过不等于页面可用，HTTP 200 或 SSR HTML 可返回也不等于客户端导航可用。页面可运行且环境允许时，应实际打开受影响页面，并尽量按用户真实入口路径进入，确认没有页面级 runtime error、白屏、错误边界、关键资源加载失败或控制台明显报错。

### 工程验证

优先运行项目已有命令：

- lint
- typecheck
- test
- build
- dev / preview（如果需要页面与路径验证）

没有这些命令时，不要强行添加新工具。说明未能验证的部分。

如果是纯 HTML / 静态文件项目：

- 不要强行寻找 lint、typecheck、build。
- 优先打开入口 HTML，例如 `index.html`，并检查相关链接或入口是否能进入受影响区域。
- 如果浏览器直接打开会影响相对路径、模块脚本或 fetch 行为，启动简单静态服务器再验证。
- 检查 console error、资源 404、白屏、样式丢失、脚本初始化失败和关键交互。
- 修改 CSS / JS 后，至少复查受影响页面和相关资源加载。

如果任一验证命令失败：

- 先判断是否由本次修改引入。
- 如果是本次修改引入，必须继续修复，直到命令通过或明确阻塞。
- 如果是既有失败，说明它是既有问题，并补充本次改动是否新增了错误。
- 不要在构建失败、页面打不开或关键路由崩溃时宣称 fix 已完成。

### 页面与路径验证

页面可运行且环境允许时，打开受影响页面、路由或组件预览，并优先验证用户真实路径。验证顺序如下：

1. 对齐验证目标：优先使用用户提供或项目文档中的启动命令、URL / 端口、路由、登录态和操作路径。不要用另一个临时端口、空状态页面、直接 URL 或不同启动模式证明修复已完成，除非能说明两者等价。如果页面可通过导航、菜单、列表、卡片、搜索结果或 CTA 进入，必须验证至少一个真实上游入口；直接访问目标 URL 只能作为补充。
2. 管理服务来源：先确认目标端口是否已有服务。如果复用用户已启动的服务，只记录 URL / 端口，不要擅自关闭；如果本次 fix 为验证临时启动 dev / preview / static server，记录启动命令、URL / 端口、启动目录和进程信息。
3. 区分路径结果：分别记录 `Direct route check`（直接打开目标 URL）、`Client navigation check`（从真实入口点击进入）、`Deep link check`（刷新或直接深链进入）。路由/页面/client component/数据渲染修复至少覆盖 direct route、client navigation 和 console/runtime error；导航/header/menu 修复至少覆盖一个入口到目标页，以及目标页到相邻页面或核心操作。
4. 检查页面可用性：至少确认没有白屏、错误边界、hydration/runtime error、关键资源 404、关键交互初始化失败和明显 console error。对 Next.js App Router、RSC、client component 和 `next/link` 路由，必须区分直接加载和客户端跳转；build 通过、HTTP 200 或 SSR HTML 可返回，都不能证明 client navigation 没有 runtime error。
5. 处理失败和轻量场景：如果页面或路径验证失败且失败来自本次修改，进入 Recovery Gate，先修页面级错误，再继续 UI 复检；如果只修样式但页面入口明确，也做一次轻量打开验证。无法打开浏览器、Playwright、CDP 或截图验证时，只能标记为浏览器验证未完成；不能用 curl、Invoke-WebRequest、HTTP 200 或 build 通过替代浏览器路径验证，并把未覆盖项列为剩余风险。
6. 保存证据：如果生成截图，优先保存到当前项目的 `examples/reports/assets/fix/` 下，按本次修复建立子目录，例如 `examples/reports/assets/fix/2026-05-25-admin-terms/`。文件名应包含页面或区域、视口、状态和阶段，例如 `terms-375-after.png`、`terms-1280-filter-open-after.png`；如果修复前也截图，使用 `before` / `after` 成对命名。修复代码本身必须写到真实项目位置，不放入 evidence 目录。除用户要求保留、输出中明确列出的截图 / 报告 / 证据产物或真实项目改动外，验证中创建的临时脚本、临时页面、临时数据和一次性下载文件应在输出前删除；无法删除时写入 `Residual risk(剩余风险)`。
7. 清理服务：验证结束后，关闭本次 fix 临时启动的服务，并检查目标端口是否仍被占用；只有用户明确要求保留运行，或关闭会影响用户已有服务时才不关闭，并在输出中说明。在 Windows、npm、Next.js、Vite、Storybook 等多进程启动场景下，不要把父进程停止等同于服务已关闭；如果端口仍被占用，只清理能确认属于本次 fix 启动进程链的进程，无法确认时不要关闭，并写入 `Dev server(开发服务)` / `Residual risk(剩余风险)`。

### UI 验证

- 复检相关视口，至少覆盖被修问题出现的宽度；页面可运行时优先用浏览器或截图确认。
- 对布局、媒体、表格、滚动、弹层和视觉 token，确认问题没有从局部转移到别处，主题方向没有被替换。
- 对按钮、表单、菜单、弹窗、导航、原生控件替换和自定义控件，复检关键状态、键盘路径、点击热区、移动端和同类页面一致性。
- 没有浏览器时，基于代码说明无法验证的布局、pointer、hover、交互和视觉风险。

## 7. Output Gate / 输出格式

```markdown
## Fix Summary / 修复摘要
- 修复了什么：
- 保留了什么：
- 是否保持可运行：
- 未处理什么：

## Gate Results / Gate 结果
- Baseline(基线):
- Source mode(来源模式):
- Scope(范围):
- Verification(验证):
- Direct route check(直接路由验证):
- Client navigation check(客户端导航验证):
- Deep link check(深链刷新验证):
- Runtime errors(运行时错误):
- Dev server(开发服务):
- Verification target(验证目标):
- Recovery(失败回修):
- Recheck(复检):
- Screenshots(截图):
- Not verified(未验证):
- Residual risk(剩余风险):

## Changed Files / 修改文件
- `path/to/file`

## Remaining Questions / 待确认问题
- ...
```

输出要具体，不要只写“优化了 UI”。说明修了哪个问题、如何修、是否复检。如果本次保存了截图，列出截图目录和关键截图文件；如果只是临时截图或无法落盘，也说明原因。没有完成真实浏览器路径验证时，必须填写 `Not verified` 和 `Residual risk`，不要省略。

如果本次为验证创建过临时文件，输出前确认它们已删除；保留的文件必须是用户要求保留、证据产物或真实项目改动，并列在 `Screenshots(截图)`、`Changed Files(修改文件)` 或 `Residual risk(剩余风险)` 中。

## 8. 禁止事项

- 不要把 fix 变成 redesign，或在用户未要求时改变主题、明暗模式、品牌方向、页面气质或 preset。
- 不要因为审美偏好重写页面、做无关重构、换技术栈、引入不必要依赖，或删除用户内容、业务逻辑、真实数据。
- 不要伪造 lint/build/browser 验证结果；不要在本次修改导致构建失败、页面打不开或关键路由崩溃时结束任务。
- 不要把本次验证临时启动的 dev / preview / static server 留在后台占用端口，除非用户明确要求保留运行。
- 不要只停止启动返回的父 PID 就声称服务已关闭；必须以端口或子进程检查作为依据。
- 不要为了修一个问题引入新的响应式、状态、可访问性或全局视觉回归。
- 不要用隐藏、裁切、删除内容来伪装修复 overflow、表格、媒体或长文案问题。
