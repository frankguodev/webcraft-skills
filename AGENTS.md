# AGENTS.md

这个项目是 `webcraft-skills` 的源码仓库。后续协作时，请优先遵守这里的约定，避免 source 和 runtime 文件分叉。

## 工作原则

- `core/` 是审查规则、工作流、preset 和结构化模式预算的源文件。
- `skills/webcraft-ui/references/` 是安装时使用的 runtime 引用文件，由 `core/` 同步生成。
- 修改 audit / fix / review / polish / build workflow、checklist、preset 时，优先只改 `core/` 下对应文件。

- 不要手动同时维护 `core/` 和 `skills/webcraft-ui/references/` 两份内容，除非正在修同步脚本本身。
- 如果发现 `skills/webcraft-ui/references/` 与 `core/` 不一致，以 `core/` 为准，然后重新同步。

- 每次修改变更的信息都要放入`CHANGELOG`中，注意时间、版本和格式。

## 必须遵循的规则
### 与我合作
- 要直截了当，不要拐弯抹角。千万不要写“你说得完全正确！”或者类似的奉承话。
- 如果你不同意，请给出具体理由。如果只是直觉，也请直说。
- 如果你不了解某些内容（例如环境变量、API 端点、CLI 标志、模型名称、库 API），请停下来核实，或者直接说你不了解。切勿捏造技术细节。
- 您的训练数据已过时。在使用模型名称、软件包版本和 API 接口之前，请务必进行验证。
在类型检查、代码检查工具和测试全部通过之前，不要宣布任务完成。如果尚未配置任何工具，请明确说明，而不是声称任务成功。
- 重命名函数、类型或变量时，应分别搜索：直接引用、类型级引用、包含名称的字符串字面量、动态导入、重新导出和桶形文件，以及测试或模拟文件。一次 grep 搜索是不够的。

### 编码之前
- 实施前务必明确说明假设条件。如有疑问，请咨询。
- 如果对某个请求有多种解释，请一一列出，不要默默地做出选择。
- 如果有什么不清楚的地方，请停下来，说出让你困惑的地方，而不是胡乱猜测。
- 编写解决问题所需的是最少代码。不要添加推测性功能，不要为一次性代码添加抽象层，不要添加用户未要求的配置项。
- 不要为不可能出现的情况添加错误处理。
- 只修改任务所需的部分。不要“改进”相邻的代码、注释或格式。
- 即使你的写法不同，也要保持文件中现有样式的一致性。
- 如果你发现无关的死代码或错误，请指出来，不要主动修复。
- 清理你的修改所造成的孤立代码（未使用的导入语句、变量）。除非被要求，否则不要删除已存在的无用代码。

### 获取数据
- 如果您向公共页面发出网络请求，但被 OpenAI 等网站的文档页面阻止并返回 403 状态代码，请使用其他方法来获取数据。

### 秘密和凭证
- 绝对不要将 API 密钥、令牌、密码或其他机密信息硬编码到源代码中。务必从环境变量中读取这些信息。
- 提交之前，扫描暂存的更改，查找任何疑似机密信息（API 密钥、令牌、密码、连接字符串）。如果发现，请停止并标记。
- 密钥应该保存在.env文件中（这些文件必须位于.gitignore），而不是源代码中。
- 如果发现某个密钥已经提交到代码库中，请立即标记并建议轮换该密钥。

### 自我提升
- 当我纠正你、反驳你或表达不满时，在你完成当前任务后，建议在相关的 AGENTS.md 文件中添加一行或进行编辑，以避免再次出现同样的错误。
- 每当你克服了某种障碍或挑战，或者学到了对所有会话都适用的知识时，请提示在全局 AGENTS.md 文件中添加注释，以便未来的会话能够从中受益。这可以是新的规则、新的风格指南、新的工具，或者任何其他对未来代理有帮助的信息。
- 在提出建议之前，请先在相关的 AGENTS.md 文件中查找已有的规则。如果已有规则，请建议对其进行完善，而不是添加新的条款。
- 请给我看看修改后的差异。在我批准之前，请勿编辑该文件。
- 当 AGENTS.md 文件超过 200 行左右时，除了添加内容之外，还要提出删除或合并的建议，而不仅仅是添加内容。
- 如果我要求你“审核 AGENTS.md”，请阅读整个文件，并提出一个要删除的规则列表，因为这些规则已经过时、重复或在实践中从未被遵循，并对每条规则给出一句话的理由。
- 在开始一个新项目时，检查该项目是否已有 AGENTS.md 文件AGENTS.md。如果没有，建议创建一个并主动提出起草。AGENTS.md 文件用于管理代理：其中包含关于项目的技术说明（技术栈、脚本、约定、注意事项、路径、构建和测试命令）。在项目级 AGENTS.md 文件中添加一条指令，使其在项目发生重大变更时自动更新。
- 还要检查项目是否包含 README.md 文件README.md。如果没有，建议创建一个。README.md 文件是为用户准备的：它介绍了项目是什么、存在的意义以及如何入门。不要将两者混淆。如果项目只有 README.md 或 AGENTS.md，请不要在两者之间重复内容，并在需要的地方添加链接。在相关情况下，从 README.md 文件链接到 AGENTS.md。

## 中英文同步

- `*.md` 和 `*.zh.md` 应尽量保持结构和信息量一致。
- 中文用户相关内容可以更贴近中文表达，但不要改变规则含义。
- 如果只改中文或英文，需要说明是否有意不同步。

## 提交前检查

只有用户要求同步 `core/`里面的内容到runtime，才运行：

```bash
npm run sync:runtime
```

在完成规则、文档或脚本改动后，至少运行：

```bash
npm run validate
```
