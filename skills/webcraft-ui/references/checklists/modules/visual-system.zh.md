# Visual System 高风险模块

用于补强 `ui-audit.zh.md` 中视觉体系的检查，重点发现 spacing、typography、color、radius、border、shadow 和装饰语言不成系统的问题。

## 使用时机

- 页面看起来像多个模板拼接，或者用户反馈“不精致”“风格乱”“AI 味重”。
- 同屏出现多套圆角、边框、阴影、按钮风格、卡片密度、图标线宽或强调色。
- fix/polish 需要保留主题，但提高成熟度和一致性。

## 必查点

- 层级系统：H1/H2/body/caption/button 的字号、字重、行高和间距是否形成少量稳定层级。
- 色彩用途：主色、强调色、状态色、链接色、数据高亮是否有明确职责，避免全屏同色或高饱和滥用。
- 容器语言：card、panel、modal、popover、table、form control 的 radius、border、shadow 是否来自同一尺度。
- 空间节奏：页面容器宽度、section spacing、卡片内边距、grid gap 和按钮间距是否成比例。
- 装饰克制：渐变、玻璃拟态、发光、背景图、纹理、mask、filter、blend mode 是否服务信息，而不是掩盖内容不足。
- 图标和媒体：图标线宽、尺寸、颜色继承、插图/mockup/截图比例是否和组件体系一致。
- 主题保留：修复时应收敛现有方向，不在未授权时换成另一种审美。

## 分级提示

- `Critical`：视觉体系问题直接导致核心内容不可读、主操作不可辨认，或主题/内容关系混乱到用户无法理解页面。
- `Major`：同屏 token、组件、色彩职责或装饰语言明显分裂，降低专业感、可信度或操作判断。
- `Minor`：局部 spacing、radius、shadow、图标线宽或动效节奏不够统一，但不破坏整体系统。

## 浏览器验证

- 截取首屏和一个内容密集区，确认同类组件是否像同一产品。
- 在桌面和移动端检查字号、间距、卡片密度、按钮层级是否仍成立。
- 对 hover/focus/active 状态检查是否沿用同一色彩和边框体系。

## 典型问题信号

- 主按钮是 pill，卡片是大圆角，输入框是小圆角，弹窗又是另一套圆角。
- 所有可强调元素都用同一个高饱和色，用户无法判断主操作。
- 每个 section 都有不同背景、卡片、图标和装饰手法，像拼贴模板。
- 阴影、玻璃拟态或发光太重，导致界面廉价或阅读疲劳。
- 修复后虽然问题少了，但主题被换掉，前后像两个产品。

## 修复边界

- 不要用“更高级”作为理由改主题；必须说明对清晰度、一致性、可用性或真实感的影响。
- 优先统一 token 和组件关系，再处理局部 polish。
- 视觉修复应保留业务内容和产品语气，不用装饰替代真实信息结构。
