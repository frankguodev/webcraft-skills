# Layout 高风险模块

用于补强 `ui-audit.zh.md` 中最容易漏掉的布局类问题，重点不是重新审美，而是检查首屏区域之间是否真的成立。

## 使用时机

- 审查 landing page、产品页、首页、搜索页、后台首页或任何首屏承担主要理解/转化任务的页面。
- 截图中出现 hero、mockup、搜索框、标签组、统计区、卡片区、下一段内容互相靠得很近或很远。
- 用户反馈“页面看着怪”“布局不对”“好像没对齐”，但没有明确指出原因。

## 必查点

- 首屏关系：标题、说明、主操作、视觉容器、搜索区域、标签组和下一段内容之间是否有清晰主次。
- 区域边界：hero、搜索区、标签区、内容区是否互相压住、断裂、错位、悬空或过度空洞。
- 构图轴线：首屏采用中心轴、左轴、双栏轴还是视觉对象轴；标题、说明、搜索框、标签组和后续内容容器是否共享同一对齐逻辑，或有足够视觉重量解释不同对齐。
- 搜索 / 列表信息页：带大搜索框的词典、资源索引、知识库和搜索页通常应让标题、说明和搜索栏围绕中心轴建立稳定入口，后续内容也应跟随同一容器节奏，而不是突然左偏、漂移或被大空白隔断。
- 视觉容器：mockup、插画、图表、截图或装饰容器的内容量是否匹配占用面积，避免像资源未加载的空壳。
- 对齐系统：页面容器宽度、左右 padding、section spacing、grid/flex gap 是否形成同一套节奏。
- 层叠关系：sticky、fixed、absolute、z-index、overlay、toast、dropdown 是否遮挡核心文本或操作。
- 内容压力：标题变长、搜索框变窄、标签数量增加、卡片数量变化时，首屏布局是否仍稳定。

## 分级提示

- `Critical`：首屏核心信息、主 CTA、导航或主要内容被遮挡、裁切、不可读、不可点击，或常见视口下布局关系崩坏。
- `Major`：布局可用但显著降低扫描效率、任务理解、空间秩序或专业感，例如首屏空洞、容器比例失衡、层级不清。
- `Minor`：局部对齐、间距、卡片高度或装饰占位不够精致，但不影响核心理解和操作。

## 浏览器验证

- 至少在 `375px` 和 `1280px` 看首屏完整截图。
- 如果页面可运行，滚动到下一段，确认首屏和下一段不是靠偶然高度拼在一起。
- 用 DevTools 或截图确认是否存在横向滚动、元素重叠、不可见占位、空白过大或控件压线。

## 典型问题信号

- 首屏视觉区很大但内容很少，像占位框或加载失败。
- 搜索框、标签组、下一段标题与 hero 挤在一起，阅读顺序不自然。
- 标题和搜索框明显想形成中心入口，但 chips、卡片组或下一段内容突然采用另一套宽度 / 对齐，导致页面看起来偏、散或不均匀。
- 为了制造呼吸感把首屏和内容区拉得过远，导致列表信息页首屏空洞、核心内容下沉或扫描路径断裂。
- 背景网格、图片、装饰层与内容层对齐不一致，造成“框住了一块但里面没东西”的感觉。
- 桌面端左右布局看似平衡，但移动端或平板端某个区域突然漂移、压住或断开。

## 修复边界

- 默认保持原主题和视觉方向，只修布局关系、比例、间距、容器宽度和层级。
- 不要因为首屏有问题就重写整页风格；先让现有元素排布成立。
- 如果视觉容器本身没有真实内容，优先缩小、补真实内容、改为轻量背景或移除，而不是换主题。
