# UI Build 工作流

用于从零生成网站、完整页面，或在现有页面中构建有明确用户任务的功能模块。`build-ui` 的目标是交付真实可用、符合项目语境、后续可维护的界面，而不是一张漂亮但脆弱的概念图。

Build 必须继承 audit、review、polish、fix 的质量标准：先理解目标和约束，再组织结构和实现，最后自检、验证并修复明显问题。

## 语言规则

读取本 workflow 前，skill 的 Locale Contract 已经选择了一个运行语言。继续使用这个语言，不要再打开配对的英文或中文 workflow/checklist/module 文件，除非用户明确要求翻译、双语比较、本地化或语言一致性检查。

## 1. Build 范围

所有 build 默认都要达到生产级基础质量：可运行、可读、可点击、响应式不崩、状态有闭环、不编造事实，并尊重现有项目体系。构建范围只决定验证深度和完整度，不降低质量底线。

- `Site Build`：从 0 到 1 构建网站、官网、产品站、多页面骨架或完整静态站点。重点是信息架构、全局导航、首页结构、关键页面入口、整体视觉系统和可扩展结构。
- `Page Build`：默认范围，用于构建一个完整页面。重点是首屏主次、页面结构、CTA、内容区块、状态、响应式、内容压力和基础可访问性。
- `Feature Build`：构建有明确用户任务的页面区域或功能模块。重点是任务闭环、入口和出口、表单/列表/操作反馈、状态、错误恢复，以及与现有页面融合。

小按钮、单个输入框、孤立卡片、图标按钮等小组件通常不需要进入 `build-ui`；直接按项目习惯实现。只有它承载明确用户任务、影响页面结构或属于一组功能模块时，才按 `Feature Build` 处理。

如果用户明确要求 prototype、quick mock 或 rough draft，可以缩小内容、状态和验证范围，但仍必须可运行、响应式可用、占位内容明确，且不编造真实业务事实。

## 2. 输入与内容真实性

信息不足时不要机械追问，也不要随意编造。

必须先问或放入 `Open Questions` 的情况：

- 页面主体、产品名、目标用户或核心任务完全不清楚。
- 用户要求高保真、上线质量或真实业务页面，但核心内容缺失。
- 是否沿用现有风格、是否指定 preset、是否允许外部资源会影响实现方向。
- 需求涉及真实价格、客户、指标、案例、合规、承诺、团队、奖项或敏感业务事实。

可以合理补全的内容：section 顺序、次级按钮文案、空状态文案、FAQ 占位、示例列表、占位图片说明、非关键辅助文案、原型阶段 mock 内容。补全必须克制、具体、可信，并标明 placeholder。

真实事实缺失时，使用中性、可替换的占位内容；Landing page 可以补信息结构和辅助说明，但不能补“证据”。产品名、价格、客户、案例、指标、合规声明、团队背景和奖项必须来自用户、现有项目或明确资料。

## 3. Build Gate

动手前先建立可交付边界，不要在入口、风格、内容来源和验证方式都模糊时直接生成大页面：

- `Scope`：Site / Page / Feature，还是不需要进入 `build-ui` 的小组件任务。
- `Entry`：新文件、新路由、现有页面插入点、功能模块挂载位置、导航入口、返回路径或示例容器。
- `System`：现有组件、路由约定、全局样式、design tokens、字体、颜色、spacing、圆角、边框、阴影和组件状态。
- `Content`：真实内容、用户提供内容、可替换 placeholder，或不能编造的业务事实。
- `Verify`：lint、typecheck、test、build、dev / preview、浏览器打开、静态 HTML 或截图检查。

用户只要功能模块，就只做模块，不顺手重做整页；用户要小组件时，通常直接按项目习惯实现；用户要原型时，不扩展成复杂产品。范围越接近真实页面、真实流程或上线使用，状态、内容压力和验证越完整。

## 4. 现有系统、参考图与 Preset

现有项目中默认沿用当前体系：目录和路由约定、组件库、token、全局样式、组件状态、页面语气和技术栈。不要为了新页面引入另一套视觉体系、组件库、preset 或依赖。

用户提供截图、设计图、Figma 导出图或参考页面时，先判断用途：

- 结构参考：参考布局、信息顺序和组件关系。
- 风格参考：参考颜色、字体感觉、圆角、边框、阴影和氛围。
- 完整设计稿：结构和风格都尽量还原。

新项目没有既有体系时，可以按参考图结构和风格生成，并补必要的响应式、可访问性和真实内容适配。现有项目中，如果用户未明确要求换风格，默认“结构参考设计图，视觉沿用现有项目”。不确定时先问一句；如果用户不想停下来确认，采用保守策略。

Preset 只有用户明确指定时才用。没有指定时，按页面类型和项目语境选择方向；现有项目有明确体系时，preset 只能作为参考，不得覆盖现有风格。不要默认套用 `cinematic-minimal`，也不要为了“高级感”强行改成暗色、电影感、极简或 SaaS 风。Preset 不是产品事实来源。

## 5. 信息结构

先规划结构，再做视觉。不要从背景、渐变、卡片、动效开始。

- 首屏：页面是谁、做什么、给谁用、下一步动作是什么。
- 主体内容：每个 section 只表达一个核心信息。
- 行动路径：主 CTA、次 CTA、联系/购买/注册/继续阅读路径。
- 信任信息：只使用真实案例、作品、功能说明、FAQ 或数据解释。
- 状态边界：loading、empty、error、success、disabled、长内容、无图片。

页面类型模板只作为起点，不机械套用：

- Landing / Product：hero、价值说明、工作流或功能、真实 proof、CTA / FAQ。
- Dashboard / Admin：overview、controls、main content、detail/action、data states。
- Portfolio / Personal：hero、selected work、process、about、contact。
- Form / Checkout / Onboarding：step context、input groups、review、submit states。
- Docs / Content：title/summary、TOC、content sections、code/table、next step。

## 6. 实现边界

新静态页面要生成完整可运行文件，CSS 组织清晰，不依赖不可用远程资源，移动端、平板、桌面都可用。

现有项目页面要遵循已有目录、路由、组件和样式方式，优先复用组件和 token，不引入新依赖，不破坏已有路由和业务逻辑。新增页面时判断是否需要导航、菜单、列表、卡片、CTA、搜索结果或上级页面入口；如果新增入口，也要提供返回列表、breadcrumb、相关页面或下一步操作。

Feature Build 要围绕任务闭环实现，不按整页标准过度设计，也不依赖父容器偶然尺寸才能正常显示。

不要默认添加登录、权限、用户系统、数据库、CMS、后端 API、支付、多语言、主题切换、动画系统、复杂状态管理、mock API 或大型筛选/批量操作系统。如果这些能力对体验必要，只做轻量 UI 占位或说明。

## 7. 质量底线

根据页面功能补齐相关状态：按钮、表单、列表/数据、弹窗/菜单、导航都应有清楚的 hover、active、focus-visible、disabled、loading、empty、error、success 或当前状态。没有相关功能时不要硬塞状态展示，但涉及用户操作的地方必须有闭环。

默认至少考虑 375px、768px、1280px，避免横向滚动、文字贴边、按钮太小、文案溢出、媒体/表格/代码块溢出、sticky/fixed 遮挡，以及桌面布局在移动端只是缩小版。对固定格式元素使用 `max-width`、`min-width: 0`、`aspect-ratio`、合理 wrapping 和断点。

生成时必须考虑真实内容变化：长标题、长按钮文案、中英文混排、不同图片比例、卡片内容长短不一、列表 0/1/3/10/20 项、长错误文案、长用户名/项目名/邮箱/文件名。不要生成只适配理想短文案的布局。

检查 AI 模板感：过多 badge、bento grid、渐变光斑、空泛口号、虚构数据、无意义图标、重复 CTA 和填空间卡片。用真实内容和清楚结构替代装饰堆叠。

## 8. 自检与验证

生成完成后先自检并直接修复明显问题：

1. Critical：不可读、不可点击、溢出、遮挡、导航/表单/弹窗不可用。
2. 是否符合现有视觉体系、参考图用途或用户指定 preset。
3. spacing、typography、颜色、圆角、边框、阴影和组件状态是否统一。
4. 移动端、长内容和真实内容压力是否会破坏布局。
5. 是否存在明显 AI 模板感或虚构事实。

验证按构建风险分层，不要把所有 build 都跑成完整 audit：

- 新静态 HTML：打开入口 HTML；如果相对路径、模块脚本或 fetch 会受影响，启动简单静态服务器验证。
- 现有项目新增页面：运行项目已有 lint / typecheck / build 中可用的命令，并打开新路由或目标页面。
- 新增导航入口：从真实入口点击进入新页面或新 section；直接打开 URL 只能作为补充。
- 新功能模块：确认模块在目标页面、示例容器或实际使用位置能渲染，并检查主要入口、状态和操作反馈。
- 纯视觉局部构建：做轻量工程验证和目标区域检查即可；小组件任务不需要升级成完整 build 验证矩阵。

如果浏览器、dev server 或截图验证失败，不能用 build 通过、HTTP 200 或文件存在替代页面可用性验证；在输出中写入 `未验证` 和剩余风险。

## 9. 何时转入其它 workflow

- 用户要求检查已有页面：转入 `review-ui` 或 `audit-ui`。
- 生成后出现页面级 runtime error、路由不可达、关键入口无法点击或构建失败：先按 `fix-ui` 修到可运行。
- 生成后只需局部细节提升：转入 `polish-ui`。
- 生成后发现系统性质量问题或用户要求上线前检查：建议 `Focused Audit` 或 `Deep Audit`。

Build 不承担所有质量工作。它负责生成可用初稿，并完成必要自检和验证。

## 10. 输出格式

```markdown
## Build Summary

- 构建了什么：
- 构建范围：
- 采用了什么结构：
- 入口 / 路由：
- 导航集成：
- 是否参考了设计图：
- 遵循了哪些现有风格或 preset：
- 占位内容：

## Changed Files

- `path/to/file`

## States Covered

- hover / active / focus-visible / disabled / loading / empty / error / success

## Responsive Checks

- 375px:
- 768px:
- 1280px:

## Verification

- 已运行：
- 未验证：
- 剩余风险：
```

不要只输出抽象设计建议。能实现时直接实现，并说明验证结果。

## 11. 禁止事项

- 不要在用户未指定 preset 时强行套用 preset。
- 不要在现有项目中把参考图风格当成默认风格，除非用户明确要求。
- 不要引入另一套视觉体系、组件库、新依赖或换技术栈，除非用户要求。
- 不要编造真实客户、价格、指标、案例、团队、奖项或业务承诺。
- 不要生成只适配理想短文案的页面。
- 不要只做静态好看的截图，忽略状态、响应式、入口路径和内容压力。
- 不要为了丰富页面增加无意义卡片、徽章、图标、渐变和动效。
- 不要把简单页面扩展成带后端、认证、数据库和复杂状态管理的大项目。
- 不要伪造 lint、build、browser 或截图验证结果。
