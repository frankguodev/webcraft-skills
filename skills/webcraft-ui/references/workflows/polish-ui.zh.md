# UI Polish 工作流

用于在保留产品含义、信息结构、技术栈和视觉方向的前提下，让已有 UI 更精致、更真实、更少 AI 模板感。Polish 是克制润色，不是 redesign。

## 语言规则

读取本 workflow 前，skill 的 Locale Contract 已经选择了一个运行语言。继续使用这个语言；除非用户明确要求翻译、双语比较、本地化或语言一致性检查，不要打开配对语言文件。

## 1. 定位

动手前先判断粗糙感主要来自哪里，不要没有诊断就到处微调：

- page type / composition：页面类型和构图目标不匹配，例如搜索 / 词典 / 资源索引页被处理成松散 landing，或标题、搜索区、内容区没有共享稳定轴线。
- spacing / rhythm：间距忽密忽空、section 缺少停顿，或为了呼吸感把相关模块拉断、让首屏内容下沉。
- typography：字号层级乱、行高压迫、标题断行不自然。
- color：颜色层级不清、强调色过度、对比度不足。
- radius / border / shadow：圆角、边框、阴影不成体系。
- states：cursor / pointer、hover、focus-visible、disabled、loading、error 缺失或粗糙。
- responsive：移动端拥挤、溢出、按钮太小、媒体比例不稳。
- template smell：文案空泛、badge 太多、装饰多于信息。

如果问题主要来自构图关系或页面类型不匹配，先让布局关系成立，再处理局部 token。

## 2. 边界

适合 polish：页面基本可用，但显得粗糙、不稳定、不够真实；需要收敛 spacing、typography、color、radius、border、shadow、状态或模板噪音；用户要求“更精致”“少点 AI 味”“优化细节”“整体太拥挤 / 呼吸感不够”。

不适合 polish：核心流程不可用、严重溢出、遮挡、不可点击、弹窗不可关闭、表单无法提交；需要重做信息架构、产品定位、核心文案、主要转化路径、跨页组件体系或视觉方向。

review / audit findings 只能作为诊断线索；执行前确认仍属于 polish 范围，且不会改变信息结构或重复同页已有内容。Critical、阻断性 Major 或影响核心任务的问题先转 `fix-ui`；跨页面系统性问题先转 `audit-ui`。用户只是说“更精致”但风险不明时，先轻量诊断。用户提供参考网站、截图或设计方向时，先盘点现有模块和参考页可迁移模块，再判断替换、合并、弱化、删除或少量新增；默认先合并再新增。只借鉴布局骨架、模块职责、元素种类、密度关系、图标语义和交互入口，不复制参考站的品牌视觉、文案、数据、指标、logo 或业务承诺；如果需要新增事实性内容、改变产品定位或重做信息架构，先停止并给 plan。

## 3. 保留与收敛

默认保留：产品含义、业务逻辑、信息结构、主要内容顺序；用户提供的真实文案、数据、价格、案例和品牌信息；现有技术栈、组件库、路由和状态逻辑；已有品牌方向、页面语气和视觉体系。

优先收敛：使用现有 token，不为 polish 新造颜色、圆角、阴影或 spacing 体系；能局部修就不重构组件；能删除噪音就不添加装饰；统一组件状态，而不是增加新的视觉效果。

Polish 默认不新增页面级信息模块、数据概览、排行榜、地图、推荐区或内容集合；需要新增这些内容时，先停止并说明已超出 polish。可以轻微调整空泛、重复、明显模板化的文案；如果会影响产品表达，放入 `Open Questions`。

## 4. 执行深度

对外只有 `polish`。不要要求用户选择 Light / Standard / Deep；根据用户目标、页面风险和改动范围自行选择执行深度。

- 轻量：单个区域、单个组件或少量同类组件；只调 spacing、对齐、局部 radius、border、hover/focus 状态。
- 标准：默认深度；用于一个页面、一个主要流程或一组紧密相关组件；统一局部视觉 token、typography、section 节奏、组件状态和响应式细节。
- 深度：页面整体质感粗糙但不需要 redesign；允许页面级收敛 spacing、字体层级、颜色层级、radius、border、shadow、共享状态和表现结构。遇到信息架构、定位、跨页系统、路由结构、新依赖或组件库替换问题时停止并给 plan 或转 workflow。

## 5. 优先级

1. 可用性边界：阻断使用的问题转 `fix-ui`。
2. 响应式细节：移动端换行、按钮高度、媒体比例、sticky 遮挡、弹窗滚动。
3. 信息层级：页面类型、首屏主次、构图轴线、section 呼吸感、标题 / 正文节奏、CTA 层级。
4. 视觉 token：spacing、font scale、color、radius、border、shadow。
5. AI 模板感：过多 badge、bento grid、渐变光斑、空泛 slogan、装饰图标。
6. 微交互：cursor / pointer、hover、active、focus-visible、transition、图标文字对齐和点击热区。

## 6. 分项策略

- Spacing：先统一页面容器、section 间距、分组关系和组件 padding；同时检查“太挤”和“太空”，避免为了呼吸感把相关模块拉断、让首屏内容下沉或降低列表扫描效率。移动端缩小间距但不要贴边，也不要把桌面密度直接压缩到小屏。
- Composition：根据页面类型选择构图目标。搜索、词典、资源索引等带大搜索框的信息页，首屏通常应形成中心轴，让标题、说明、搜索框、热门标签和后续内容容器共享清楚的对齐逻辑；居中不是机械居中，还要检查视觉重量、宽度、留白和下一段露出是否均匀适中。后台、dashboard 或高密度列表可保持左轴，但需要足够内容密度和明确分组支撑。
- Content Structure：检查 section 是否职责重复、同类卡片网格是否连续堆叠、卡片面积是否匹配信息量。同类内容只能有一个主表达；重复出现必须承担不同职责、权重或形态，否则优先合并、弱化或删除。目录 / 搜索 / 词条 / 资源索引页的短内容入口不应默认使用大 padding、大高度的大卡片；只有标题、一句描述和一个状态时，应使用紧凑列表、小入口卡、分组入口、趋势列表或降低视觉权重。可以在不改变产品事实的前提下，把重复模块整理成分类入口、趋势列表、场景入口、紧凑更新列表、完整列表或方法论说明，让页面形成产品路径而不是卡片堆积。
- Typography：减少层级数量，控制字号 / 字重跳跃，改善 line-height、段落宽度和标题换行。
- Color：保留品牌色，限制强调色范围，先保证文字对比度和状态可识别。
- Radius / Border / Shadow：收敛小型 token scale；边框用于结构，阴影只用于真实层级。
- Components / States：补齐真实可点击元素的 cursor / pointer、hover、active、focus-visible、disabled、loading、empty、error、success；状态反馈不应改变元素尺寸、相邻布局或滚动位置。
- Icons / Click targets：统一图标尺寸、线宽、文字基线和图标文字间距；图标必须服务分类、状态或动作识别，同屏重复使用同一图标时要确认是否表达同一语义；图标按钮和紧邻操作的点击 / 触控热区要足够清楚。
- AI Template Smell：优先删除、弱化或收敛无意义 badge、装饰图标、重复 CTA 和填空间卡片；只有不重复同页已有信息且服务当前任务时，才补少量辅助内容。
边界不清时，按需参考 `layout`、`visual-system`、`components-states`、`responsive`、`forms-controls` 和 `ai-template-smell` 模块；如果判断结果已经影响核心任务、可访问性或数据操作，停止 polish，转 `fix-ui` 或 `audit-ui`。

## 7. 复检

润色后检查：

- 375px、768px、1280px 关键视口，以及长标题、长按钮文案、中英文混排、长标签、badge、chip 和错误文案。
- 真实可点击元素的 cursor / pointer、hover、active、focus-visible、disabled、loading、error。
- 图标与文字对齐、图标线宽和尺寸、图标按钮热区、弹窗、菜单、表单和导航可用性。
- 视觉 token 是否更统一，是否引入新的不一致。
- 页面是否更容易扫描，section 是否有停顿但不过分空洞，主要 CTA 是否没有被同权重元素淹没。
- 首屏标题、说明、搜索区、标签组、卡片组或下一段内容是否处在同一构图逻辑中；搜索 / 列表信息页要特别检查中心轴、容器宽度和首屏内容露出。
- 每个 section 是否回答一个清楚问题、服务一个明确入口或决策；不能有两个 section 用相同形态回答同一个入口问题。
- 是否仍保留原页面风格和产品含义，且没有把同页已有列表、统计、分类、标签或 CTA 复制到新的装饰区或摘要区。

页面可运行且环境允许时，打开目标页面做轻量复检。对齐用户实际入口；如果本次 polish 临时启动服务，记录命令、URL / 端口和进程信息，复检结束后关闭并检查端口；复用用户已有服务时只记录，不擅自关闭。生成截图时放入 `examples/reports/assets/polish/<polish-run>/`；临时对照文件、草稿、验证脚本和临时数据默认在输出前删除。

## 8. 输出

输出包含：`Source(来源)`、`Boundary(边界)`、`Preserved(保留)`、`Adjusted(调整)`、`Removed(删除)`、`Changed Files(修改文件)`、`Verification(验证)`、`Remaining Questions(待确认问题)`。说明具体收敛了哪些视觉细节、删除了哪些模板噪音、是否保留原风格，以及哪些内容未验证。

## 9. 禁止事项

- 不要把 polish 变成 redesign，或在用户未指定 preset 时强行套用 preset。
- 不要默认改成暗色、电影感、极简或 SaaS 风。
- 不要为了高级感增加渐变、光斑、卡片、动效。
- 不要删除真实业务信息、重写产品定位或核心文案，除非用户要求。
- 不要把参考站当成视觉皮肤照抄，也不要复制参考站的品牌、文案、数据、指标、logo 或业务承诺。
- 不要引入新依赖、换组件库、重写路由或改变业务逻辑。
- 不要用隐藏 overflow、裁切内容、缩小到不可读或删除必要信息来制造“看起来更干净”的假 polish。
- 不要只靠压缩间距、缩小字号或塞满卡片制造“整齐”，也不要只靠放大 section padding、拉开模块或稀释内容密度制造“高级感”。
- 不要伪造浏览器验证结果，不要把本次验证临时启动的服务留在后台，也不要关闭 polish 开始前已存在或无法确认属于本次 polish 的服务。
