# Forms And Controls High-Risk Module

用于补强 `ui-audit.zh.md` 中表单、筛选、搜索、选择控件和批量操作的检查，重点发现“能看见但不敢操作、能提交但不知道结果”的问题。

## 使用时机

- 页面包含登录、注册、上传、编辑、结算、搜索筛选、后台批量操作、危险操作或设置表单。
- 项目已有自定义控件，但页面仍混用突兀的原生 `select`、checkbox、radio、file input 或未封装输入框。
- 用户反馈表单“不像一套系统”“不知道哪里错了”“提交后没反馈”。

## 必查点

- 标签和上下文：输入框、选择器、文件上传、开关和批量操作必须有可理解的 label 或等价上下文，不能只靠 placeholder。
- 状态闭环：idle、focus、disabled、loading、error、success、empty 必须覆盖核心路径。
- 错误恢复：错误文案要说明问题和修复方式，并和对应控件在视觉/语义上关联。
- 控件体系：input、select、combobox、dropdown、multi-select、checkbox、radio、switch、textarea、file upload 是否沿用同一视觉和交互系统。
- 原生控件适配：appearance、accent-color、caret-color、color-scheme、resize、field-sizing 等能力是否和产品风格匹配。
- 移动端操作：键盘、错误提示、提交按钮、底部操作条、弹窗表单不能互相遮挡。
- 数据密度：筛选器、批量操作、长选项、多选标签、长错误文案和空状态要能承载真实内容。

## 浏览器验证

- 用键盘走一遍核心表单路径，确认焦点顺序、focus-visible、提交、错误和恢复。
- 在移动宽度检查输入框、键盘区域、错误提示和提交按钮是否互相压住。
- 如果页面可运行，至少触发一次 loading/disabled 或错误状态；不能运行时从代码确认状态分支。

## 典型问题信号

- placeholder 消失后用户不知道输入内容含义。
- 表单提交按钮可重复点击，没有 loading 或 disabled。
- 错误只显示红色边框，没有文案或没有关联到具体字段。
- 后台筛选区里自定义按钮旁边混入浏览器默认 select，显得像另一个产品。
- 多选标签过多后挤爆工具栏，或长选项撑宽弹层。

## 修复边界

- 优先修可完成性、错误恢复和状态闭环，再修视觉统一。
- 如果已有组件库或自定义基础控件，优先复用；不要给单个页面写孤立控件风格。
- 不要为了视觉简洁删除必要 label、帮助文本、错误文本或确认信息。
